# `erc8004/` — Identity, Registration & the Health Gate

Everything that makes Vouch discoverable as an ERC-8004 agent on GOAT Network, plus the deployment gate that refuses to register an unhealthy agent.

```
erc8004/
├── registration.json         The ERC-8004 registration document (what gets pinned)
├── abi.ts                    Identity + Reputation registry ABIs (viem-typed)
└── scripts/
    ├── healthcheck.ts        Pre-registration health gate
    ├── pin-registration.ts   Pin registration.json to IPFS (Pinata)
    └── register.ts           On-chain registration on GOAT testnet3
```

---

## registration.json — field guide

| Field | What it is |
|-------|------------|
| `type` | Always `https://eips.ethereum.org/EIPS/eip-8004#registration-v1` |
| `name`, `description` | Vouch + the full pitch: what it scores, who calls it, how it settles. This is the text agents/marketplaces see |
| `image`, `homepage` | Rewritten to `VOUCH_PUBLIC_URL` at pin time — placeholders are fine in the repo copy |
| `category`, `tags` | Discovery keywords (agentic-payments, wallet-scoring, sybil-detection, kya, x402, erc-8004 …) |
| `capabilities` | Machine-readable list of what Vouch can do (wallet_financial_scoring, pre_payment_gating, sybil_detection …) |
| `services` | The five endpoints (below) |
| `x402Support` | `true` — and it must be real: the health check verifies a live 402 |
| `registrations` | `agentRegistry` (CAIP-10 of the Identity Registry) + `agentId` (filled in after registering) |
| `supportedTrust` | `reputation`, `crypto-economic` |

### Services

| Service | Endpoint (after rewrite) | Purpose |
|---------|--------------------------|---------|
| `x402` | `/v1/score` | The paid scoring API |
| `MCP` | `/.well-known/mcp.json` | Tool manifest: `score_wallet`, `quick_score` with input schemas, x402 auth block, examples |
| `A2A` | `/.well-known/agent-card.json` | Agent card: skills, payment scheme, capabilities |
| `WEB` | `/` | Free discovery surface with pricing + metrics |
| `OASF` | (spec repo) | Skill/domain taxonomy entries |

The server serves the same document live at `/.well-known/agent.json` with endpoints rewritten to the current `VOUCH_PUBLIC_URL` — the health check enforces that the pinned copy and the live copy advertise the same services.

## Deployment workflow (health-gated)

```
deploy server → npm run healthcheck → npm run pin → npm run register → update agentId
```

### 1. `npm run healthcheck`

**Input:** `VOUCH_PUBLIC_URL` (the live deployment to probe).
**Output:** PASS/FAIL per check; exit 0 only when all pass.

| # | Check | Fails when |
|---|-------|-----------|
| 1 | `/health` returns `healthy` | server down, `VOUCH_PAY_TO_ADDRESS` unset, or dev-skip-payment mode on |
| 2 | `/` discovery | `x402Support` or pricing missing |
| 3 | `/.well-known/agent.json` | not an eip-8004 doc |
| 4 | `/.well-known/mcp.json` | `score_wallet` / `quick_score` tools missing |
| 5 | `/.well-known/agent-card.json` | x402 auth scheme missing |
| 6 | `POST /v1/score` → real 402 | paywall disabled or challenge/payTo missing |
| 7 | live registration endpoints | any service endpoint points at another host or a placeholder |
| 8 | live vs static parity | services list in repo differs from what's served |

### 2. `npm run pin`

**Input:** `PINATA_JWT`, `VOUCH_PUBLIC_URL` (must not be localhost).
**What it does:** canonicalizes `image`/`homepage`/service endpoints to the public URL (same rewrite as the server), pins to IPFS via Pinata with CIDv1.
**Output:** the content-addressed `ipfs://<cid>` URI to use as `AGENT_URI`.

### 3. `npm run register`

**Input:** `AGENT_URI` (`ipfs://` preferred), `VOUCH_PRIVATE_KEY` (testnet3 gas), `VOUCH_PUBLIC_URL`.
**What it does:** re-runs all health checks (abort on any failure; escape hatch `SKIP_HEALTHCHECK=true`), then calls `register(agentURI)` on the Identity Registry and parses the `Registered` event.
**Output:** tx hash, explorer link, and the assigned `agentId`.

### 4. After registration

Copy the printed `agentId` into `registration.json` → `registrations[0].agentId`, re-pin, and update the on-chain URI with `setAgentURI(agentId, newURI)` (owner key required).

## Registry addresses

| Network | Chain ID | Identity Registry | Reputation Registry |
|---------|----------|-------------------|---------------------|
| GOAT Testnet3 | 48816 | `0x556089008Fc0a60cD09390Eca93477ca254A5522` | `0xd9140951d8aE6E5F625a02F5908535e16e3af964` |
| GOAT Mainnet | 2345 | `0x8004A169FB4a3325136EB29fA0ceB6D2e539a432` | `0x8004BAa17C55a88189AE136b182e5fdA19dE9b63` |

Agent IDs are **per chain** — a registration on another network does not carry over to GOAT. Testnet3 uses different registry addresses than mainnet (mainnet matches the canonical CREATE2 deployments).

`abi.ts` also exports the Reputation Registry `giveFeedback(agentId, value, valueDecimals, tag1, tag2, endpoint, feedbackURI, feedbackHash)` ABI for the roadmap attestation feature (`/v1/attest`).
